### 方式二：S_simple 小项目流程

**适用场景**：单一数据对象、单一模块

| 判断条件 | 说明 |
|---------|------|
| 单一数据对象 | 如 User、Article、Comment 等 |
| 单一模块可解决 | 不需要跨模块协作 |
| API 端点少 | 2-5 个 API 端点 |
| 开发周期短 | 1-3 天可完成 |

#### 文档阶段（S0-S3，含测试设计）

```
S0_init.txt
    ↓
S1_features.txt → S2_schema.txt → S3_api.txt
    ↓
⚠️ 上下文重置（清空会话）
```

| 步骤 | 文件 | 生成内容 | 耗时 |
|------|------|----------|------|
| S0 | `S_simple/初始化/S0_init.txt` | 项目结构、配置文件 | 5分钟 |
| S1 | `S_simple/文档阶段/S1_features.txt` | docs/FEATURES.md + 测试策略 | 8分钟 |
| S2 | `S_simple/文档阶段/S2_schema.txt` | docs/SCHEMA.md + 数据层测试 | 8分钟 |
| S3 | `S_simple/文档阶段/S3_api.txt` | docs/API_DESIGN.md + API测试 | 10分钟 |

#### 实施阶段（S4-S10，每步有测试报告）

| 步骤 | 文件 | 生成内容 | 耗时 |
|------|------|----------|------|
| S4 | `S_simple/实施阶段/S4_infra.txt` | main.py、数据库配置、依赖 | 10分钟 |
| S5 | `S_simple/实施阶段/S5_models.txt` | 数据模型代码 + 测试 + 报告 | 15分钟 |
| S6 | `S_simple/实施阶段/S6_repo.txt` | Repository 代码 + 测试 + 报告 | 15分钟 |
| S7 | `S_simple/实施阶段/S7_service.txt` | Service 代码 + 测试 + 报告 | 15分钟 |
| S8 | `S_simple/实施阶段/S8_api.txt` | API 代码 + 测试 + 报告 | 15分钟 |
| S9 | `S_simple/实施阶段/S9_verify.txt` | Streamlit前端 + E2E测试 | 10分钟 |
| S10 | `S_simple/实施阶段/S10_report.txt` | 最终测试报告 | 10分钟 |

**注意**：S4-S10 在同一个会话中完成，保持上下文连贯。

### 方式三：L_large 大项目流程

**适用场景**：多数据对象、多模块协作

| 判断条件 | 说明 |
|---------|------|
| 多数据对象 | User、Order、Product、Payment 等需要关联 |
| 多模块协作 | 需要拆分为多个独立模块 |
| API 端点多 | 超过 5 个 API 端点 |
| 开发周期长 | 超过 3 天 |

#### 第一步：整体规划（L0-L7）

| 步骤 | 文件 | 生成内容 | 耗时 |
|------|------|----------|------|
| L0 | `L_large/初始化/L0_init.txt` | 项目结构（多模块） | 10分钟 |
| L1 | `L_large/文档阶段/L1_lane.txt` | docs/ARCHITECTURE.md | 15分钟 |
| L2 | `L_large/文档阶段/L2_prd.txt` | docs/PRD.md（整体） | 10分钟 |
| L3 | `L_large/文档阶段/L3_features.txt` | docs/FEATURES.md（整体） | 10分钟 |
| L4 | `L_large/文档阶段/L4_business.txt` | docs/BUSINESS.md | 10分钟 |
| L5 | `L_large/文档阶段/L5_schema.txt` | docs/SCHEMA.md（整体） | 10分钟 |
| L6 | `L_large/文档阶段/L6_api.txt` | docs/API_DESIGN.md（整体） | 10分钟 |
| **L7** | `L_large/文档阶段/L7_split.txt` | docs/MODULE_SPLIT.md | 15分钟 |

**⚡ L7 是关键节点**：定义模块边界，为后续模块开发做准备

#### 第二步：模块实施（对每个模块调用 S_simple）

```
L7 模块拆分 → 读取 MODULE_SPLIT.md
    ↓
对每个模块执行 S_simple 流程
    ↓
模块A → S_simple（14步）
模块B → S_simple（14步）
模块C → S_simple（14步）
    ↓
⚠️ 模块之间需要上下文重置
```

#### 第三步：组合测试（L14-L21）

| 步骤 | 文件 | 生成内容 | 耗时 |
|------|------|----------|------|
| L14 | `L_large/实施阶段/L14_infra.txt` | 基础设施配置 | 10分钟 |
| L15 | `L_large/实施阶段/L15_models.txt` | 各模块数据模型 | 15分钟 |
| L16 | `L_large/实施阶段/L16_repo.txt` | 各模块Repository | 15分钟 |
| L17 | `L_large/实施阶段/L17_service.txt` | 各模块Service | 15分钟 |
| L18 | `L_large/实施阶段/L18_api.txt` | 各模块API | 15分钟 |
| L19 | `L_large/实施阶段/L19_integrate.txt` | 模块集成 | 15分钟 |
| L20 | `L_large/实施阶段/L20_gateway.txt` | API Gateway | 10分钟 |
| L21 | `L_large/实施阶段/L21_verify.txt` | 端到端验证 + 报告 | 20分钟 |

## 上下文重置说明

### 需要重置的节点

| 节点 | 原因 |
|------|------|
| S3 → S4 | 文档阶段→实施阶段，对话较长 |
| L7 → 模块A | 整体规划→模块实施 |
| 模块A → 模块B | 避免跨模块上下文污染 |

### 不需要重置的节点

| 节点 | 原因 |
|------|------|
| S4 → S5 → ... → S10 | 同一实施阶段，保持上下文 |

## 常见问题

### Q: 小项目可以用 L_large 吗？

A: 可以，但不推荐。L_large 会产生更多文档，小项目用 S_simple 效率更高。

### Q: 大项目可以用 S_simple 吗？

A: 不推荐。大项目如果用 S_simple，会导致代码耦合、难以维护。

### Q: 什么时候需要上下文重置？

A: 只在文档阶段→实施阶段（S3→S4）之间，以及切换模块时需要重置。实施阶段内部（S4-S10）不需要重置。

### Q: 支持哪些编程语言？

A: 提示词基于 Python/FastAPI 设计，但可适配其他语言。
