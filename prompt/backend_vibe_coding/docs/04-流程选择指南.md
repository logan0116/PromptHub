## D 快速通道详细说明

### D1: Bug 修复

**何时使用**：
- 修复已知的错误或缺陷
- 解决报错问题
- 修复功能异常

**流程特点**：
- 定位问题 → 确认方案 → 执行修复 → 验证 → 提交
- 不生成文档，直接修复代码
- 通常 5-8 步完成

**示例**：
- "用户登录时返回 500 错误"
- "数据分页不正确"
- "邮箱验证不生效"

---

### D2: 配置变更

**何时使用**：
- 修改环境变量
- 调整数据库配置
- 修改中间件设置
- 更新日志配置

**流程特点**：
- 定位配置文件 → 确认变更 → 修改配置 → 验证生效 → 提交
- 不修改业务代码

**示例**：
- "更换数据库连接地址"
- "调整 Redis 缓存配置"
- "启用 CORS 跨域"

---

### D3: 小优化

**何时使用**：
- 优化查询性能
- 重构代码结构
- 添加缓存
- 改进并发处理

**流程特点**：
- 分析代码 → 确认优化方案 → 执行优化 → 性能测试 → 提交
- 关注性能和代码质量

**示例**：
- "优化用户列表查询速度"
- "添加 Redis 缓存"
- "重构重复代码"

---

### D4: 文档更新

**何时使用**：
- 更新 API 文档
- 补充代码注释
- 更新 README
- 完善架构文档

**流程特点**：
- 定位文档 → 确认更新内容 → 更新文档 → 格式验证 → 提交
- 只修改文档，不修改代码

**示例**：
- "更新 API 文档"
- "补充函数注释"
- "更新部署说明"

## S_simple 小项目流程详细说明

### 适用场景

**满足以下任一条件即可**：

| 条件 | 说明 |
|------|------|
| 单一数据对象 | 如 User、Article、Comment 等 |
| 单一模块可解决 | 不需要跨模块协作 |
| API 端点少 | 2-5 个 API 端点 |
| 开发周期短 | 1-3 天可完成 |

### 流程概览

```
文档阶段（S0-S6，约 30-45 分钟）
├── S0: 项目初始化（5分钟）
├── S1: 需求文档 PRD（5分钟）
├── S2: 功能梳理（5分钟）
├── S3: API 设计（5分钟）
├── S4: 数据模型设计（5分钟）
├── S5: 服务层设计（5分钟）
└── S6: 测试设计（5-10分钟）
        ↓
    ⚠️ 上下文重置（清空 AI 会话）
        ↓
实施阶段（S7-S13，约 60-90 分钟）
├── S7: 基础设施（10分钟）
├── S8: 数据模型代码 + 测试（15分钟）
├── S9: Repository 代码 + 测试（15分钟）
├── S10: Service 代码 + 测试（15分钟）
├── S11: API 代码 + 测试（15分钟）
├── S12: 验证测试（10分钟）
└── S13: 最终测试报告（10分钟）
```

### 项目示例

**适合 S_simple 的项目**：
- 用户管理系统
- 文章发布系统
- 评论系统
- 标签管理系统
- 简单的 CRUD 系统

## L_large 大项目流程详细说明

### 适用场景

**满足以下任一条件即可**：

| 条件 | 说明 |
|------|------|
| 多数据对象 | User、Order、Product、Payment 等需要关联 |
| 多模块协作 | 需要拆分为多个独立模块 |
| API 端点多 | 超过 5 个 API 端点 |
| 开发周期长 | 超过 3 天 |
| 复杂业务流程 | 涉及多个业务场景 |

### 流程概览

```
第一步：整体规划（L0-L7，约 60-90 分钟）
├── L0: 项目初始化（10分钟）
├── L1: 泳道图 + 时序图（15分钟）
├── L2: 整体需求文档（10分钟）
├── L3: 整体功能梳理（10分钟）
├── L4: 业务文档（10分钟）
├── L5: 整体数据模型（10分钟）
├── L6: 整体 API 设计（10分钟）
└── L7: 模块拆分（15分钟）⚡ 关键节点
        ↓
第二步：单一模块（调用 S_simple，每个模块约 90-120 分钟）
├── 模块A → 执行完整的 S_simple 流程
├── 模块B → 执行完整的 S_simple 流程
└── 模块C → 执行完整的 S_simple 流程
        ↓
第三步：组合测试（L19-L21，约 30-45 分钟）
├── L19: 模块集成（15分钟）
├── L20: API Gateway（10分钟）
└── L21: 端到端验证（20分钟）
```

### 关键节点：L7 模块拆分

**L7 是大项目流程的核心节点**，它决定了：
- 模块如何划分
- 模块之间如何通信
- 模块开发的先后顺序
- 模块如何集成

### 项目示例

**适合 L_large 的项目**：
- 电商系统（用户、商品、订单、支付、物流）
- ERP 系统（采购、库存、销售、财务）
- 社交平台（用户、内容、关系、消息、通知）
- 在线教育（课程、学员、教师、订单、证书）

## 对比总结

| 维度 | D 快速通道 | S_simple 小项目 | L_large 大项目 |
|------|-----------|-----------------|----------------|
| 适用场景 | Bug、配置、优化、文档 | 单一模块新功能 | 多模块复杂系统 |
| 步骤数 | 5-8 步 | 14 步 | 22+ 步 |
| 文档生成 | 否 | 是（完整设计文档） | 是（整体 + 模块文档） |
| 测试要求 | 视情况而定 | 每层独立测试 | 单元 + 集成 + E2E |
| 耗时 | 5-30 分钟 | 1.5-2.5 小时 | 3+ 小时 |
| 上下文重置 | 不需要 | S6→S7 重置一次 | 多次重置 |

## 使用建议

1. **从 A_Decision.txt 开始**
   - 让 AI 帮你自动判断合适的流程
   - 避免手动选择的困扰

2. **小项目不要强行用 L_large**
   - 单一模块用 S_simple 效率更高
   - L_large 适用于真正的多模块系统

3. **快速通道不要滥用**
   - 新功能开发不要用 D 快速通道
   - D 仅适用于 Bug 修复、配置变更等小改动

4. **注意上下文重置**
   - S_simple 在 S6→S7 时需要重置
   - L_large 在 L7→L14 和切换模块时需要重置
