### 1. GSD（Get-Shit-Done）借鉴

#### 1.1 轻量级通道（GitLab Issue 思想）

**GSD 的做法**：
- 大改动走完整 Discuss → Plan → Execute → Verify
- 小改动直接做，不走完整流程

**我们已实现**：
```
大改动：A → B1~6 → C1~6（完整 13 步）
小改动：D（直接搞定，5-8 步）
```

**为什么**：
- 修复一个 bug 不需要重新生成 PRD、API_DESIGN 等
- 改个配置不需要走完整流程
- **效率提升**：小改动从 30 分钟降到 5 分钟

---

#### 1.2 上下文重置机制

**GSD 发现的问题**：
```
传统 AI Coding：会话越长，AI 越糊涂
- 0-30% 上下文：质量优秀
- 30-50% 上下文：质量下降
- 50%+ 上下文：质量断崖下跌
```

**GSD 的解决方案**：
- 阶段完成后清空上下文
- 新阶段重新加载必要文档
- 每步都有"干净的脑子"

**我们要做的**：
```
B6 完成后 → 清空会话 → C1 重新开始
```

**为什么**：
- 你的流程有 13 步，B 阶段 6 步，C 阶段 6 步
- B→C 时对话已经很长了
- **AI 开始"脑补"**：忘记之前的约定、自己加东西
- **重置后质量稳定**

---

#### 1.3 原子提交

**GSD 的做法**：
- 每个任务完成后立即 commit
- 每个 commit 都是独立的、可回滚的

**我们要做的**：
```
C1 完成 → git commit "feat(models): 添加数据模型层"
C2 完成 → git commit "feat(repo): 添加 Repository 层"
C3 完成 → git commit "feat(service): 添加 Service 层"
C4 完成 → git commit "feat(api): 添加 API 层"
```

**为什么**：
- 如果 C2 有问题，只需要回滚 C2，不影响 C1
- **git bisect** 能精确定位哪个 commit 有问题
- 代码历史清晰，便于后续 AI 理解

---

### 2. ClaudeCodeSpecWorkflow 借鉴

#### 2.1 EARS 需求格式

**ClaudeCodeSpecWorkflow 的做法**：
```
需求：用户登录

WHEN 用户提交登录请求
- THEN 系统 SHALL 验证邮箱格式
- IF 邮箱无效 THEN 系统 SHALL 返回 400 错误
```

**我们要做的**：
- 在 B1_PRD.txt 中加入 EARS 格式
- AI 生成的需求可以直接转代码

**为什么**：
- 自然语言太模糊："系统应该能区分新老用户"
- EARS 格式精确："IF 用户已关联业务账户 THEN 系统 SHALL 判定为老用户"
- **AI 直接按 EARS 写代码，不会"脑补"**

## 第三阶段：架构重构（v6.0 大版本）

**目标**：重构为分层架构，支持小项目/大项目/模块化

### 背景

经过实际使用和思考，发现现有架构存在以下问题：

1. **单一流程不适合所有场景**：小项目被强制走完整流程，效率低
2. **大项目缺少模块拆分能力**：无法将大项目拆分为独立微服务
3. **快速通道过于笼统**：Bug 修复、配置变更、小优化应该有不同的提示词
4. **缺少多智能体基础**：无法为未来多智能体系统打基础

### 新架构设计

```
┌─────────────────────────────────────────────────────────────┐
│                    A_Decision.txt（入口）                     │
│  1. 判断任务类型（新功能 / 快速修复）                          │
│  2. 判断项目规模（小项目 / 大项目）                            │
│  3. 初始化基础结构                                            │
└──────────────────────┬──────────────────────────────────────┘
                       │
           ┌───────────┴───────────┐
           ▤                       ▤
    ┌─────────────┐         ┌─────────────┐
    │   快速修复   │         │   新功能开发  │
    │  D1~D4      │         │  → 继续判断   │
    └─────────────┘         └──────┬──────┘
                                   │
                       ┌───────────┴───────────┐
                       ▤                       ▤
                ┌─────────────┐         ┌─────────────┐
                │   小项目     │         │   大项目     │
                │  S1 → S6    │         │  三步走流程   │
                └─────────────┘         └──────┬──────┘
                                               │
                    ┌──────────────────────────┼──────────────────────────┐
                    ▤                          ▤                          ▤
            ┌───────────────┐          ┌───────────────┐          ┌───────────────┐
            │  ① 整体规划   │          │  ② 单一模块   │          │  ③ 组合测试   │
            │  L1 泳道图     │    ┌────→│  调用 S1~S6   ├────┐    │  L 集成验证    │
            │  L2 需求文档   │    │    │  逐模块生成    │    │    │  API Gateway  │
            │  L2.5 模块拆分 │    │    └───────────────┘    │    └───────────────┘
            └───────────────┘    │                          │
```

### 核心理念

**小项目**：单一数据对象、单一模块可解决 → 直接走 S1~S6

**大项目**：必须拆分 → 三步走
1. **整体规划**：先看全局（泳道图、需求文档、模块拆分）
2. **单一模块**：每个模块调用小项目流程（S1~S6）生成
3. **组合测试**：模块集成、API Gateway、整体验证
