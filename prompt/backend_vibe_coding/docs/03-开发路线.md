### 1. GSD（Get-Shit-Done）借鉴

#### 1.1 轻量级通道（GitLab Issue 思想）

**GSD 的做法**：
- 大改动走完整 Discuss → Plan → Execute → Verify
- 小改动直接做，不走完整流程

**我们已实现**：
```
大改动：A → B1~6 → C1~6（完整 13 步）
小改动：D（直接搞定，5-8 步）
```

**为什么**：
- 修复一个 bug 不需要重新生成 PRD、API_DESIGN 等
- 改个配置不需要走完整流程
- **效率提升**：小改动从 30 分钟降到 5 分钟

---

#### 1.2 上下文重置机制

**GSD 发现的问题**：
```
传统 AI Coding：会话越长，AI 越糊涂
- 0-30% 上下文：质量优秀
- 30-50% 上下文：质量下降
- 50%+ 上下文：质量断崖下跌
```

**GSD 的解决方案**：
- 阶段完成后清空上下文
- 新阶段重新加载必要文档
- 每步都有"干净的脑子"

**我们要做的**：
```
B6 完成后 → 清空会话 → C1 重新开始
```

**为什么**：
- 你的流程有 13 步，B 阶段 6 步，C 阶段 6 步
- B→C 时对话已经很长了
- **AI 开始"脑补"**：忘记之前的约定、自己加东西
- **重置后质量稳定**

---

#### 1.3 原子提交

**GSD 的做法**：
- 每个任务完成后立即 commit
- 每个 commit 都是独立的、可回滚的

**我们要做的**：
```
C1 完成 → git commit "feat(models): 添加数据模型层"
C2 完成 → git commit "feat(repo): 添加 Repository 层"
C3 完成 → git commit "feat(service): 添加 Service 层"
C4 完成 → git commit "feat(api): 添加 API 层"
```

**为什么**：
- 如果 C2 有问题，只需要回滚 C2，不影响 C1
- **git bisect** 能精确定位哪个 commit 有问题
- 代码历史清晰，便于后续 AI 理解

---

### 2. ClaudeCodeSpecWorkflow 借鉴

#### 2.1 EARS 需求格式

**ClaudeCodeSpecWorkflow 的做法**：
```
需求：用户登录

WHEN 用户提交登录请求
- THEN 系统 SHALL 验证邮箱格式
- IF 邮箱无效 THEN 系统 SHALL 返回 400 错误
```

**我们要做的**：
- 在 B1_PRD.txt 中加入 EARS 格式
- AI 生成的需求可以直接转代码

**为什么**：
- 自然语言太模糊："系统应该能区分新老用户"
- EARS 格式精确："IF 用户已关联业务账户 THEN 系统 SHALL 判定为老用户"
- **AI 直接按 EARS 写代码，不会"脑补"**
