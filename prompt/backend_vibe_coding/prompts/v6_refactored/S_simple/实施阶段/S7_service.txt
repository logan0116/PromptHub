# 重要原则

1. **一步一步来**：每次只做一个任务，完成后立即暂停，等待我的输入
2. **必须提供选项**：所有需要我做出决定的问题，都必须使用 question 工具提供选项
3. **不要假设**：不要自行假设或填写任何内容

# 使用 question 工具

当需要我做出选择时，必须调用 question 工具。选项应该根据项目实际情况灵活设计：
- 提供合理的预设选项
- 提供清晰的选项描述
- **默认支持多选**，除非该问题天然是单选（如确认类问题）

## 重要：多选必须设置 multiple 参数

需要多选的问题必须在问题对象中添加 `"multiple": true`，否则只能单选。

### 正确示例（多选）

根据上下文（业务逻辑、服务设计等）动态生成选项：

```
{
  "questions": [
    {
      "header": "测试场景",
      "question": "请选择需要覆盖的测试场景（可多选）：",
      "multiple": true,
      "options": [
        {"label": "正常场景", "description": "成功业务流程"},
        {"label": "参数验证", "description": "输入参数校验"},
        {"label": "异常场景", "description": "错误处理流程"},
        {"label": "边界条件", "description": "边界值测试"}
      ]
    }
  ]
}
```

注意：不要添加"自定义"选项，question 工具会自动提供"自定义"输入框。

# 步骤 9：生成测试报告

运行测试后，手动创建 Markdown 测试报告到 `tests/reports/S7_services_report.md`，包含：

**报告格式示例**：
```markdown

**生成日期**: YYYY-MM-DD
**测试范围**: services
**测试结果**: ✅ 全部通过

---

## 测试摘要
| 指标 | 数值 |
|------|------|
| 总测试数 | XX |
| 通过 | XX |
| 失败 | XX |
| 跳过 | XX |
| 执行时间 | X.XXs |

---

## 详细结果
| 测试用例 | 状态 |
|----------|------|
| test_create_xxx | ✅ PASSED |
| test_validate_input | ✅ PASSED |

---

## 覆盖的 Service 方法
| 方法 | 状态 | 说明 |
|------|------|------|
| create_xxx | ✅ | 创建业务对象 |
| validate_xxx | ✅ | 参数验证 |
| process_xxx | ✅ | 业务处理 |
```

将生成的报告路径告知用户，并询问是否继续。
